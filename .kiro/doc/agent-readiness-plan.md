# Agent-Readiness Improvement Plan

> Goal: Improve stormacq.com score on isitagentready.com (currently 67)
> Constraints:
> - Minimize cost — prefer free AWS features and native Hugo capabilities
> - All CloudFront changes must be done via CDK in `scripts/cdk-codebuild/`
> Status: **Deployed** (2026-04-26)

## Architecture

- **Static site**: Hugo (PaperMod theme), built with `hugo --minify`
- **Hosting**: S3 bucket `stormacq.com` + CloudFront distribution `E3MEG4UBSP7M9Y` (Free pricing plan)
- **Build pipeline**: AWS CodePipeline → CodeBuild (`buildspec.yaml`) → S3 deploy
- **CloudFront Functions**: Two functions managed via CDK:
  - `stormacq-viewer-request` (viewer-request) — URI rewriting + Accept header content negotiation
  - `stormacq-viewer-response` (viewer-response) — Link headers + Content-Type override for .md files
- **Lambda@Edge (origin-response)**: `scripts/lambda@edge 20240724.js` — handles redirects for `/about`, `/podcasts`, and legacy 403 fallback to `/old/`
- **llms.txt**: generated by Hugo template in `themes/PaperMod/layouts/_default/llms.txt`
- **CDK stack**: `scripts/cdk-codebuild/app.ts` — manages CodePipeline, CodeBuild, CloudFront distribution, and CloudFront Functions

## Fix 1: Link Response Headers (RFC 8288) — DONE

### Problem
No `Link` HTTP response headers on the homepage.

### Solution: Viewer-response CloudFront Function
A **viewer-response CloudFront Function** (`stormacq-viewer-response`) injects the `Link` header on every response.

### Decision log
- **Response Headers Policy was rejected** — CloudFront's Free pricing plan does not support custom Response Headers Policies. The deploy failed with: `"Distributions with the Free pricing plan can't have the following features: Custom response headers policy"`.
- **Fallback**: viewer-response CloudFront Function. CloudFront Functions are supported on the Free plan and cost nothing for the traffic levels of a personal blog (free tier: 2M invocations/month).

### Headers added
```
Link: </llms.txt>; rel="describedby", </feed.xml>; rel="alternate"; type="application/rss+xml"
```

### Verification
```bash
curl -sI https://stormacq.com/ | grep -i '^link:'
```

## Fix 2: Markdown Content Negotiation — DONE

### Problem
When an agent sends `Accept: text/markdown`, the site returned HTML.

### Solution — three parts

| Part | What | How |
|------|------|-----|
| A | Generate `.md` files at build time | Hugo custom output format (`[outputFormats.Markdown]` in `hugo.toml`) |
| B | Route `Accept: text/markdown` to `.md` files | Viewer-request CloudFront Function (`stormacq-viewer-request`) |
| C | Set `Content-Type: text/markdown` on `.md` responses | Viewer-response CloudFront Function (`stormacq-viewer-response`) |

### Decision log
- **Part C approach changed** — Originally planned to fix S3 object metadata at upload time via a PostDeploy CodeBuild stage. Dropped in favor of the viewer-response CloudFront Function (which was already needed for Fix 1). This eliminated the need for an extra CodeBuild project and pipeline stage, reducing cost and complexity.

### Part A: Hugo Markdown output format

Added to `hugo.toml`:
```toml
[outputFormats.Markdown]
  mediaType = "text/markdown"
  baseName = "index"
  isPlainText = true

[outputs]
  home = ["HTML", "RSS", "Markdown"]
  page = ["HTML", "Markdown"]
  section = ["HTML", "RSS", "Markdown"]
```

Layout templates created:
- `layouts/_default/single.md` — renders `.RawContent` (original Markdown source)
- `layouts/_default/list.md` — section pages as Markdown link lists
- `layouts/index.md` — homepage as Markdown with 20 latest posts

### Part B: Viewer-request CloudFront Function

`scripts/cloudfront-function-request.js` — replaces the old `prod-index-rewrite` function. Checks `Accept` header for `text/markdown` and rewrites URI to `index.md` instead of `index.html`.

### Part C: Viewer-response CloudFront Function

`scripts/cloudfront-function-response.js` — overrides `Content-Type` to `text/markdown; charset=utf-8` for `.md` responses. Same function also handles Fix 1 (Link headers).

### Verification
```bash
curl -sI -H "Accept: text/markdown" https://stormacq.com/ | grep -i content-type
# Expected: content-type: text/markdown; charset=utf-8

curl -s -H "Accept: text/markdown" https://stormacq.com/ | head -20
# Expected: Markdown content, not HTML
```

## CDK and CloudFront Distribution

### Distribution imported into CDK
The CloudFront distribution `E3MEG4UBSP7M9Y` was not managed by any CloudFormation stack (created manually). It was imported into the `SebInTheCloudBuildV2` stack using `cdk import` as a `CfnDistribution` L1 construct.

### Two-phase deployment
1. **Phase 1** — `cdk import` with the distribution config matching the live state exactly (including the old `prod-index-rewrite` function ARN). CloudFormation adopted the resource without modifying it.
2. **Phase 2** — `cdk deploy` with the updated config: new CloudFront Functions wired in, Response Headers Policy removed. CloudFormation updated the distribution in place.

### Post-deployment cleanup
The old CloudFront Function `prod-index-rewrite` (ARN: `arn:aws:cloudfront::<REDACTED>:function/prod-index-rewrite`) is no longer associated with the distribution and can be deleted from the console.
